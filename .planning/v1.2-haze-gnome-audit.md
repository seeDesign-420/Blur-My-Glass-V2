# v1.2 Audit vs `haze-gnome`

Scope: compare the current `v1.2` overlay implementation in `blur-my-glass` with the similar overlay/surface handling in `/home/thomas/Coding Projects/blur-my-glass/haze-gnome`.

## Summary

- Findings: 3
- High: 1
- Medium: 2

`haze-gnome` is materially more robust in surface discovery and lifecycle handling. The current `v1.2` implementation is serviceable for the listed targets, but it is still heuristic-driven and more likely to miss reparenting/animation edge cases.

## Findings

| Severity | Finding | Evidence | Impact | Recommendation |
|---|---|---|---|---|
| High | Popup and menu discovery is brittle for dynamic shell actors. | `extension/src/components/overlays.js:178-230`, `702-804` relies on direct `Main.panel.statusArea` lookups plus style-class scans and `child-added`/`monitors-changed` only. In `haze-gnome`, `extension/lib/surfaceRegistry.js:62-81`, `156-333` rebuilds explicit popup/actor surfaces from lookup functions and refreshes when actor identities change; `extension/lib/controllers/popup.js:31-99` binds menu open-state, geometry, and ancestor signals. | Date menu, quick settings, desktop menus, and app menus can silently fail to attach or remain attached to stale actors when the shell re-parents or replaces menu containers. This directly threatens `OVR-01`, `OVR-02`, `OVR-05`, and `OVR-06`. | Replace heuristic actor scanning with explicit surface discovery/rebuild semantics for each overlay target. At minimum, add `child-removed` handling and a refresh path that re-resolves menu actors when the actor tree changes. |
| Medium | Overlay teardown happens too aggressively during close transitions. | `extension/src/components/overlays.js:377-395` detaches immediately when the target becomes unmapped/invisible, with no close-animation grace period. `haze-gnome/extension/lib/controllers/base.js:172-228` and `extension/lib/controllers/popup.js:154-260` keep the blur alive through open/close animation state, scene repaint tracking, and delayed hide ownership. | Brief mapped/visible oscillations during popup close can cause blur popping/flicker, especially for notifications, OSD, and menu surfaces that animate through transient visibility states. This is a lifecycle risk for `LIFE-01` and `LIFE-02`. | Add an explicit close-state or short hide timeout per overlay target, and keep geometry/paint tracking alive until the shell animation has actually settled. |
| Medium | `v1.2` has a single shared overlay profile, while `haze-gnome` supports per-surface tuning and adaptive brightness. | `extension/schemas/org.gnome.shell.extensions.blur-my-shell.gschema.xml:164-239` exposes one `overlays` profile with shared sigma/brightness/corner-radius/refraction settings and per-target on/off toggles. In contrast, `haze-gnome/extension/lib/runtime/settings.js:502-642` and `extension/prefs/pages/surfaces.js:1-160` support per-surface sigma, brightness, corner radius, blur engine/style overrides, and adaptive brightness. | The current defaults must fit date menu, quick settings, notifications, OSD, desktop menus, and app menus all at once. That is workable, but it makes mixed bright/dark backgrounds harder to tune and leaves no per-surface escape hatch if one surface looks wrong. This is a quality gap against Haze, though not strictly a v1.2 scope violation. | If `v1.2` stays baseline-only, document this as intentional scope. If parity with Haze is desired, add per-surface brightness/tuning or at least adaptive brightness for the overlay profile. |

## Notes

- The current `v1.2` code is cleaner than the earlier overlay state because stale states are now removed when a descriptor no longer matches, but it still does not match Haze's registry-level rebuild model.
- I did not treat Haze-only surfaces such as clipboard, looking glass, modal dialogs, or panel-menu subclasses as defects here unless they overlap with the `v1.2` targets.
- Net: `blur-my-glass` is on the right path for the listed overlay targets, but `haze-gnome` shows a more production-hardened model for actor discovery and lifecycle control.
